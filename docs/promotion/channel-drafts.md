## Show HN Draft

Title:

```text
Show HN: loop-crafter - a Codex skill for designing gated agent loops
```

Body:

```text
I built loop-crafter, a review-first Codex skill for turning recurring agent workflows into explicit loop designs before they become scaffolds, automation, or repository changes.

It focuses on practical failure points I keep seeing in long-running AI coding work: scope drift, unclear evidence, weak validation, missing state, and approval boundaries that are easy to blur.

The project is early and document-driven. It can help draft loop designs, scaffold proposals, readiness reports, and validation harness designs. It is not an unattended runner and does not publish, deploy, commit, push, tag, or release by default.

I would especially like feedback from maintainers using Codex or other coding agents on real repository work: where does the workflow feel too heavy, and what gate is missing?
```

## Reddit Draft

```text
I am preparing an open-source Codex skill called loop-crafter.

The problem it targets: long-running AI coding tasks often drift when the workflow is not explicit. A task starts as "keep going until done" and later the agent is mixing scope, evidence, validation, approval, and publication boundaries.

loop-crafter is meant to design the loop before execution: goal, trigger, evidence, roles, allowed actions, forbidden actions, validation, state, budget, and stop conditions. It can draft scaffold proposals and readiness reports, but it is not an unattended runner.

I am looking for critique from people already using AI coding agents in real repositories. What would make this useful instead of just more process?
```
