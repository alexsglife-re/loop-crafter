# Readiness Model

Use this reference before assigning readiness to a loop proposal.

Readiness describes how much action the loop may safely take. It is not authorization to execute, write files, commit, push, deploy, publish, or run unattended.

## Levels

### L0 Draft

Design only.

- Documents the goal, trigger, evidence, roles, boundary, validation, and state.
- May include unresolved decisions.
- Must not mutate source files or external systems.

### L1 Report

Report-only loop.

- Reads allowed evidence and reports findings.
- May recommend state shape, validation, or next actions.
- Must not write source files or external systems by default.

### L2 Assisted

Bounded owner-approved assistance.

- May propose scaffold files or a small bounded action.
- Requires explicit owner authorization before writing.
- Requires validation before and after any write.
- Requires PM/Advisor/Reviewer gates when project rules require them.

### L3 Unattended

Not generated by default.

- Requires proven denylist controls, budget limits, observability, verifier strength, rollback/recovery, and human escalation gates.
- Must be treated as future requirements unless the owner explicitly asks for unattended design and the governing workflow permits it.

## Block Conditions

Block, downgrade, or keep the loop at L0/L1 when any of these are true:

- Evidence allowlist is unclear.
- Validation is missing, weak, or purely aspirational.
- State location or stale-state handling is unclear.
- The loop touches denylisted paths or domains.
- The loop would require secrets, credentials, auth, permission, payment, schema, production, or destructive actions.
- The loop would publish, deploy, release, tag, force-push, rewrite history, or create irreversible external effects.
- PM/Advisor/Reviewer gates are required but missing.
- Owner authorization is required but absent.

## Missing-Control Checklist

Report missing controls in these categories:

- Evidence allowlist
- Forbidden path/action list
- Human gates
- PM/Advisor/Reviewer governance
- Deterministic verifier or rubric
- Pass/fail evidence
- Retry and budget limits
- State and run-log contract
- Secret/copy-content scan expectation
- Recovery and stop conditions

## Recommendation Vocabulary

Use one of:

- `proceed`: controls are sufficient for the target readiness.
- `revise`: non-blocking gaps should be fixed before writing or operating.
- `block`: P0/P1 or missing authorization prevents the target level.
- `keep L1`: report-only operation is appropriate; assisted or unattended operation is not justified.
